Итерация 1: Ресурсы

Проблема: всё блокирует рендеринг

  • <script> без defer блокирует парсинг HTML — браузер останавливает всё, пока скрипт не загрузится и не выполнится
  • Шрифты: font-display: block — невидимый текст 3 секунды
    (FOIT — Flash of Invisible Text)
  • Нет preload — браузер сам угадывает приоритеты,
    и угадывает плохо
⏱ 40 сек. «Первая проблема — ресурсы грузятся как попало. Script без defer — классика: парсер HTML встречает тег скрипта и стоит, ждёт. Шрифты по умолчанию блокируют отрисовку текста на 3 секунды. И нет preload — браузер вынужден сам разбираться, что важнее». Переход: «Как это починить?»

Решение: правильный <head>

❌ До

<link rel="stylesheet"
      href="/assets/main.css">
<script src="/assets/app.js"></script>
<!-- Шрифт грузится из CSS --
     браузер узнаёт о нём поздно -->
✅ После

<link rel="preload" href="/hero.jpg"
      as="image">
<link rel="preload"
      href="/fonts/Inter.woff2"
      as="font" crossorigin>
<link rel="stylesheet"
      href="/assets/main.css">
<script defer
        src="/assets/app.js"></script>
<!-- + font-display: swap в CSS -->
defer — скрипт грузится параллельно, выполняется после парсинга HTML
preload — подсказка браузеру: «это понадобится, начни загрузку сейчас»
⏱ 50 сек. «Слева — типичный head. Скрипт без defer, шрифт где-то в CSS. Справа — добавляем preload для hero-картинки и шрифта, defer для скрипта, font-display: swap в CSS. Defer не блокирует парсер, preload позволяет браузеру начать загрузку раньше. Важно: defer vs async — defer гарантирует порядок выполнения, async — нет». Переход: «Что это дало в цифрах?»

Итерация 2: Картинки

Проблема: все картинки грузятся сразу

  • 20+ картинок начинают грузиться одновременно — конкурируют за bandwidth
  • Hero-картинка (LCP element) конкурирует с остальными — браузер не знает, что она важнее
  • background-image в CSS — браузер не может оптимизировать приоритет
  • Нет width / height — Layout Shift при загрузке
⏱ 40 сек. «Все картинки начинают грузиться сразу. Hero-картинка, которая нужна пользователю первой, стоит в очереди наравне с карточками товаров внизу страницы. Плюс нет размеров — контент прыгает при загрузке». Переход: «Решение проще, чем кажется».

Решение: приоритеты и lazy loading


<!-- LCP-картинка — грузить ПЕРВОЙ -->
<img src="hero.jpg"
     fetchpriority="high"
     width="1200" height="600"
     alt="...">

<!-- Остальные — только при скролле -->
<img src="product.jpg"
     loading="lazy"
     width="300" height="300"
     alt="...">
fetchpriority="high" — говорит браузеру: «это самая важная картинка, грузи первой»
loading="lazy" — не грузить, пока пользователь не доскроллит
width + height — браузер заранее знает размер, нет layout shift
⏱ 50 сек. «Два атрибута делают разницу. fetchpriority=high для hero — браузер ставит её в начало очереди. loading=lazy для остального — не грузим, пока не нужно. И обязательно width/height — это убирает CLS полностью. Важно: fetchpriority — это хинт, не гарантия. Но браузеры его хорошо поддерживают». Переход: «Результаты вас удивят».

Итерация 3: Рендеринг

Проблема: браузер рендерит всё подряд

  • CSS-файл блокирует первый рендер
    (render-blocking resource — пока CSS не загружен, экран пустой)
  • Браузер считает layout для элементов за пределами экрана — впустую тратит время
  • Изменение одного элемента пересчитывает layout соседей — каскадный reflow
⏱ 40 сек. «Третья проблема — рендеринг. Весь CSS блокирует отрисовку — пока файл не загружен целиком, браузер показывает белый экран. Плюс браузер рассчитывает layout для всех элементов, даже тех, что внизу страницы и не видны. Critical Rendering Path — классическая тема». Переход: «Как оптимизировать рендеринг?»

Решение: критический CSS и containment

Inline critical CSS

<style>
  /* Только above-the-fold */
  .hero { ... }
  .header { ... }
</style>
<link rel="stylesheet"
      href="/main.css"
      media="print"
      onload="this.media='all'">
CSS Containment

/* Не рендерить невидимое */
.product-card {
  content-visibility: auto;
  contain-intrinsic-size: 0 300px;
}

/* Изолировать reflow */
.reviews-section {
  contain: content;
}

content-visibility: auto — браузер пропускает рендеринг невидимых элементов.
contain: content — reflow внутри блока не затрагивает соседей.

⏱ 50 сек. «Два приёма. Первый — inline критический CSS в head, остальной CSS грузим асинхронно через трюк с media=print. Второй — content-visibility: auto говорит браузеру: если элемент не виден, не рендери его вообще. А contain: content изолирует reflow — изменения внутри блока не затрагивают остальную страницу». Переход: «Что дал рендеринг в цифрах?»

Итерация 4: Сервер

Проблема: медленная доставка

  • HTTP/1.1: head-of-line blocking, максимум 6 параллельных соединений
  • Нет сжатия — файлы летят в полном размере
  • Нет кеша — каждый визит как первый
  • Нет Early Hints — сервер молчит, пока обрабатывает запрос, браузер простаивает
⏱ 40 сек. «Серверная сторона. HTTP/1.1 — максимум 6 соединений, остальные ждут в очереди. Нет сжатия — полный вес файлов. Нет кеша — каждый визит грузит всё заново. И нет Early Hints — пока сервер думает, браузер стоит без дела». Переход: «Решаем в два шага».

HTTP/2 + Brotli

HTTP/2

  • Мультиплексирование — все запросы через одно TCP-соединение
  • Бинарное фреймирование — эффективнее текстового HTTP/1.1
  • Параллельные потоки — нет head-of-line blocking

Brotli

  • На 20–30% эффективнее gzip
  • Встроенный словарь для веб-контента (HTML, CSS, JS)
  • Quality 4 — баланс скорости сжатия и размера
⏱ 50 сек. «HTTP/2 — мультиплексирование. Вместо 6 параллельных соединений — неограниченное количество потоков через одно. Brotli — современное сжатие, на 20-30% лучше gzip. У него встроенный словарь частых веб-паттернов. Quality 4 — хороший баланс между скоростью сжатия и размером файла. Для статики можно и 11, но это медленнее». Переход: «Теперь Early Hints и кеширование».

Cache-Control стратегия

Тип файла Заголовок Логика
HTML no-cache Всегда проверять свежесть
JS/CSS (с хешем) immutable, max-age=31536000 Кешировать навсегда
Картинки max-age=86400 Кешировать на сутки
⏱ 50 сек. «103 Early Hints — сервер ещё думает над ответом, но уже отправляет подсказку: начни грузить CSS и JS. Браузер не простаивает. В визуализации видно — с Early Hints загрузка ресурсов начинается раньше. Cache-Control — три стратегии: HTML всегда проверяем (no-cache), JS/CSS с хешем кешируем навсегда (immutable), картинки — на сутки». Переход: «Финальные цифры!»

Чеклист: применить в понедельник

  • ✅ Добавьте defer к скриптам и preload для критических ресурсов
  • ✅ fetchpriority="high" для LCP-картинки, loading="lazy" для остальных
  • ✅ Inline critical CSS, content-visibility: auto для длинных списков
  • ✅ Включите HTTP/2 и Brotli на сервере
  • ✅ Настройте Cache-Control для разных типов файлов

Всё это работает в любом фреймворке: React, Vue, Angular, Svelte, и даже без фреймворка.
