## Итерация 1: Загрузка ресурсов

**Ветка:** `iter-1-resources`
**Изменённые файлы:** `index.html`, `public/fonts/inter/inter.css`
**Lighthouse:** Score 71 -> 73, FCP 2711 -> 1960 мс (-28%)

### Объяснение проблемы

#### Как работает HTML-парсер

Браузер обрабатывает HTML последовательно, от первого байта до последнего. Процесс:

1. **Tokenizer** -- разбивает поток байтов на токены: открывающий тег, текст, закрывающий тег.
2. **Tree construction** -- строит DOM-дерево из токенов.
3. **Параллельно** строится CSSOM (CSS Object Model) из загруженных стилей.
4. **Render Tree** = DOM + CSSOM. Только после этого начинается layout и paint.

**Parser-blocking ресурсы** -- это точка, где весь процесс останавливается.

#### Что такое parser-blocking

Когда парсер встречает тег `<script>` **без атрибутов `defer` или `async`**:

```
HTML-парсинг идёт...
  ↓
Парсер видит <script src="...">
  ↓
СТОП. Парсинг HTML приостановлен.
  ↓
Браузер скачивает скрипт (может занять секунды на медленном соединении)
  ↓
Браузер выполняет скрипт
  ↓
Парсинг HTML продолжается
```

**Почему это критично**: всё, что находится в HTML ниже тега `<script>`, не будет разобрано и не попадёт в DOM, пока скрипт не скачается и не выполнится. Пользователь видит белый экран или частично загруженную страницу.

В нашем baseline Leaflet.js загружался как blocking-скрипт в `<head>`:

```html
<!-- ANTI-PATTERN: блокирует парсинг ВСЕГО HTML -->
<script src="/vendor/leaflet/leaflet.js"></script>
```

Leaflet нужен только для карты в самом низу страницы About, но он блокирует рендеринг всей страницы.

### defer vs async vs blocking

#### Blocking (без атрибута)

```
Парсинг HTML ──────■■■■■■■■■■■■■────────── Парсинг HTML
                   ↑ скачивание  ↑ выполнение
                   Парсер ждёт...
```

Парсер полностью останавливается. Самый худший вариант. Именно так работал Leaflet в baseline.

#### async

```
Парсинг HTML ──────────────────────■■■─── Парсинг HTML
                   ↗ скачивание ↗     ↑ выполнение
              (параллельно)         Парсер приостанавливается
```

Скачивание идёт параллельно парсингу -- это хорошо. Но **выполнение происходит сразу по завершении загрузки**, прерывая парсер. Порядок выполнения **не гарантирован** -- если у вас два скрипта с `async`, они выполнятся в порядке завершения загрузки, а не в порядке записи в HTML.

**Когда использовать**: аналитика (Google Analytics), рекламные скрипты, виджеты -- любые скрипты, которые **не зависят от других скриптов и не нужны для рендеринга страницы**.

#### defer

```
Парсинг HTML ────────────────────────────── DOM готов ──■■■
                   ↗ скачивание ↗                       ↑ выполнение
              (параллельно)                        (после полного
                                                    парсинга HTML)
```

Скачивание параллельно парсингу. Выполнение **после полного построения DOM**. Порядок выполнения **гарантирован** -- совпадает с порядком в HTML.

**Когда использовать**: почти всегда для прикладных скриптов. Если `app.js` зависит от `vendor.js`, оба с `defer` выполнятся в правильном порядке.

**Что мы сделали**: добавили `defer` на Leaflet:

```html
<!-- defer — скрипт скачивается параллельно, выполняется после DOM -->
<script defer src="/vendor/leaflet/leaflet.js"></script>
```

### Preload и приоритеты браузера

#### Preload scanner

У браузера есть **два парсера**:

1. **Основной парсер** (Main Parser) -- строит DOM, может быть заблокирован скриптами.
2. **Preload scanner** (Speculative Parser) -- работает параллельно, когда основной парсер заблокирован. Сканирует HTML вперёд и ищет ресурсы для предзагрузки.

Preload scanner находит:
- `<link rel="stylesheet" href="...">`
- `<script src="...">`
- `<img src="...">`

Preload scanner **НЕ находит**:
- URL-ы из `background-image` в CSS (не видит содержимое CSS-файлов)
- Ресурсы, загружаемые из JavaScript (`fetch()`, динамические `import()`)
- Элементы, которые ещё не существуют в HTML (создаются через JS)

#### `<link rel="preload">`

Если ресурс критически важен, но preload scanner не может его найти заранее -- или вы хотите начать загрузку раньше, чем парсер дойдёт до него, -- используйте `preload`:

```html
<link rel="preload" href="/images/hero/hero-1.jpg" as="image">
```

Браузер начинает загрузку **немедленно**, не дожидаясь, пока парсер дойдёт до `<img>` в теле страницы.

**Что мы сделали**: добавили preload для hero-картинки (LCP-элемент):

```html
<!-- Preload LCP-картинки hero-слайдера -->
<link rel="preload" href="/images/hero/hero-1.jpg" as="image">
```

#### Разница между preload, prefetch и preconnect

| Директива | Когда скачивает | Для чего |
|-----------|----------------|----------|
| `preload` | Сейчас, для текущей страницы | Критические ресурсы: LCP-картинка, главный шрифт |
| `prefetch` | В фоне, для будущей навигации | Ресурсы следующей страницы |
| `preconnect` | Устанавливает соединение | DNS + TCP + TLS к стороннему домену |

#### Приоритеты в Chromium

Chromium назначает каждому ресурсу приоритет:

| Приоритет | Ресурсы |
|-----------|---------|
| Highest | Основной CSS, шрифты (font-display: block) |
| High | Скрипты в `<head>`, preload-ресурсы, изображения выше фолда |
| Medium | Остальные скрипты |
| Low | Изображения ниже фолда, `async`-скрипты |
| Lowest | `prefetch`-ресурсы |

### font-display

#### FOIT (Flash of Invisible Text) -- `font-display: block`

```
Страница загрузилась, текст НЕВИДИМ (до 3 секунд)
  ↓
Шрифт загрузился -> текст появляется
```

Браузер ждёт до 3 секунд загрузки шрифта. Всё это время текст на странице **невидим**. Если шрифт не загрузился за 3 секунды -- показывается системный шрифт. Так работал наш baseline (`font-display: block` в `inter.css`).

#### FOUT (Flash of Unstyled Text) -- `font-display: swap`

```
Страница загрузилась, текст ВИДЕН (системный шрифт)
  ↓
Шрифт загрузился -> происходит замена шрифта (swap)
```

Текст виден **с первого момента** в системном шрифте. Когда кастомный шрифт загрузится -- происходит замена. Пользователь может начать читать сразу.

#### `font-display: optional`

Ещё лучше для перформанса: если шрифт не загрузился за ~100 мс, браузер использует системный шрифт и **не меняет его потом**. Нет layout shift, нет мерцания. При повторном визите шрифт берётся из кеша и отображается сразу.

**Что мы сделали**: заменили `font-display: block` на `font-display: swap`:

```css
/* Было: font-display: block — текст невидим до загрузки шрифта */
/* Стало: font-display: swap — текст виден сразу в системном шрифте */
@font-face {
  font-family: 'Inter';
  font-display: swap;
  src: url('/fonts/inter/inter-cyrillic.woff2') format('woff2');
}
```

### Порядок тегов в `<head>`

Порядок имеет значение. Браузер обрабатывает `<head>` последовательно. Оптимальный порядок:

```html
<head>
  <!-- 1. Meta-теги (не загружают ресурсы) -->
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>TechStore</title>

  <!-- 2. Preload критических ресурсов (начать загрузку как можно раньше) -->
  <link rel="preload" href="/images/hero/hero-1.jpg" as="image">

  <!-- 3. Стили (render-blocking, но необходимые) -->
  <link rel="stylesheet" href="/fonts/inter/inter.css">
  <link rel="stylesheet" href="/vendor/leaflet/leaflet.css">

  <!-- 4. Скрипты с defer (не блокируют, скачиваются параллельно) -->
  <script defer src="/vendor/leaflet/leaflet.js"></script>
</head>
```

В baseline порядок был хаотичным: сначала Leaflet CSS, потом blocking Leaflet JS (который блокировал всё ниже), потом шрифты. Мы переупорядочили.

### Объяснение метрик

- **FCP (First Contentful Paint)**: -28%. Текст и первые элементы появляются на ~750 мс раньше. Причины: `defer` разблокировал парсер, `font-display: swap` убрал FOIT, `preload` ускорил загрузку hero-картинки.
- **LCP без изменений (~11.3 с)**: LCP-элемент -- это hero-картинка. Preload начинает её загрузку раньше, но при Slow 3G она всё равно конкурирует за bandwidth с 20+ другими картинками. Нужна приоритизация картинок (итерация 2).
- **SI (Speed Index)**: -28%. Страница визуально заполняется быстрее.

### Возможные вопросы аудитории

**В: "Почему не `async` вместо `defer`?"**

О: `async` не гарантирует порядок выполнения. Если `app.js` зависит от `vendor.js` и оба `async`, `app.js` может выполниться раньше `vendor.js` (если он меньше по размеру и загрузится быстрее) -- и всё сломается. `defer` гарантирует порядок, совпадающий с порядком тегов в HTML.

**В: "Зачем `preload`, если браузер и так найдёт ресурсы?"**

О: Preload scanner не видит `background-image` в CSS и динамические запросы из JS. Кроме того, preload поднимает приоритет ресурса -- браузер начнёт загрузку раньше, даже если preload scanner его обнаружит. Для LCP-элемента это критично.

**В: "`font-display: optional` же не покажет кастомный шрифт?"**

О: При первом визите -- возможно, нет (если шрифт не успеет загрузиться за ~100 мс). Но он закешируется и отобразится при всех последующих визитах. Для перформанс-критичных сайтов (e-commerce, медиа) это лучший выбор, потому что гарантирует CLS = 0. Мы использовали `swap` как компромисс -- пользователь всегда видит текст, а swap малозаметен.

## Итерация 2: Оптимизация картинок

**Ветка:** `iter-2-images`
**Изменённые файлы:** `HeroSlider.vue`, `ProductCard.vue`, `ReviewsList.vue`
**Lighthouse:** Score 73 -> 80, LCP 11334 -> 4851 мс (-57%)

### Объяснение проблемы

Картинки -- это обычно 50-70% веса страницы. В нашем baseline:

- **Все 20+ картинок загружаются одновременно**: карточки товаров, аватарки отзывов, все слайды hero-слайдера -- браузер ставит их все в очередь.
- **Hero-картинка (LCP) конкурирует** с 20 другими картинками за bandwidth. При Slow 3G каждая картинка ~200-500 КБ, и они забивают канал.
- **Hero использует `background-image`**: `<div :style="{ backgroundImage: url(...) }">` -- preload scanner не может обнаружить URL картинки, пока не будет скачан и разобран CSS. Это задерживает LCP ещё сильнее.
- **Нет width/height**: при загрузке картинки контент сдвигается вниз (CLS).

### fetchpriority -- как это работает

#### Система приоритетов Chromium

Chromium назначает каждому запросу приоритет в сетевом стеке. Для картинок по умолчанию:

| Условие | Приоритет |
|---------|-----------|
| Картинка в viewport (выше фолда) | High |
| Картинка ниже фолда | Low |
| LCP-кандидат | High (но не Highest) |

Проблема: приоритет High -- это то же, что у обычных скриптов. Ресурс с Highest приоритетом (основной CSS) всегда выиграет у High.

#### Атрибут `fetchpriority`

```html
<!-- Повышаем приоритет до Highest — наравне с основным CSS -->
<img src="/images/hero/hero-1.jpg" fetchpriority="high">

<!-- Понижаем приоритет — для декоративных картинок -->
<img src="/images/decorative.jpg" fetchpriority="low">
```

`fetchpriority="high"` сигнализирует браузеру: **"Этот ресурс критически важен, загрузи его в первую очередь"**. Браузер повышает приоритет до Highest.

**Критически важно**: используйте `fetchpriority="high"` максимум на 1-2 картинках (только LCP-элемент). Если всё "высокоприоритетное" -- ничего не приоритетное.

**Что мы сделали**: переписали hero-слайдер с `background-image` на `<img>` и добавили `fetchpriority`:

```html
<!-- Первый слайд (LCP) — максимальный приоритет, eager загрузка -->
<img
  :src="slide.image"
  :fetchpriority="index === 0 ? 'high' : 'low'"
  :loading="index === 0 ? 'eager' : 'lazy'"
  width="1280"
  height="500"
>
```

Остальные слайды получают `fetchpriority="low"` и `loading="lazy"` -- они загрузятся только когда пользователь переключит слайд.

### loading="lazy" -- как работает

#### Механизм внутри браузера

Нативный `loading="lazy"` использует Intersection Observer API, встроенный в движок браузера:

1. Браузер видит `<img loading="lazy">`.
2. **Не** начинает загрузку немедленно.
3. Рассчитывает **threshold distance** (пороговое расстояние) от viewport. Это расстояние зависит от типа соединения:
   - Slow 3G: ~3000 px до viewport (начинает загрузку заранее, потому что скорость низкая)
   - Обычный 4G: ~1250 px до viewport
   - Wi-Fi: ~1250 px
4. Когда элемент приближается к viewport на threshold distance -- начинается загрузка.

**Важное правило**: НИКОГДА не ставьте `loading="lazy"` на LCP-элемент. Это **задерживает** его загрузку, ухудшая LCP. Правило:

| Позиция | Атрибут |
|---------|---------|
| Выше фолда (первый экран) | `loading="eager"` (по умолчанию) |
| Ниже фолда | `loading="lazy"` |

**Что мы сделали**: добавили `loading="lazy"` на карточки товаров и аватарки отзывов:

```html
<!-- Карточка товара — ниже фолда, lazy загрузка -->
<img
  :src="product.image"
  loading="lazy"
  decoding="async"
  width="400"
  height="300"
/>
```

`decoding="async"` -- бонус: говорит браузеру декодировать картинку в фоновом потоке, не блокируя main thread.

### Явные размеры (width/height) и CLS

#### Без размеров

```
[Заголовок]     [Заголовок]       [Заголовок]
[Текст    ]     [Текст    ]       [         ]
[Футер    ]     [         ]  -->  [Картинка  ]  --> CLS!
                [Картинка  ]      [Текст     ]
                [Футер    ]       [Футер     ]
```

1. Браузер рендерит DOM, картинка занимает 0 px (размер неизвестен).
2. Картинка загрузилась, браузер узнаёт её размеры.
3. Контент ниже картинки сдвигается вниз -- **layout shift**.

#### С размерами

```html
<img src="..." width="400" height="300">
```

Браузер **заранее** резервирует место 400x300 px в layout. Когда картинка загрузится -- она заполнит зарезервированное место без сдвига.

#### Формула CLS

```
Layout Shift Score = Impact Fraction x Distance Fraction
```

- **Impact Fraction** -- какая доля viewport затронута сдвигом.
- **Distance Fraction** -- на какую долю viewport сместился контент.

Порог: CLS > 0.1 считается плохим.

С явными width/height: CLS = 0 (сдвигов нет).

### `<img>` vs `background-image`

| Свойство | `<img>` | `background-image` |
|----------|---------|-------------------|
| Preload scanner | Да, находит URL | Нет, URL в CSS-файле |
| `fetchpriority` | Да | Нет |
| `loading="lazy"` | Да | Нет (нужен JS) |
| Accessibility (alt) | Да | Нет |
| Responsive (`srcset`) | Да | Нет |
| Семантика | Контентная картинка | Декорация |

**Правило**: всегда используйте `<img>` для контентных картинок. `background-image` -- только для чистой декорации (текстуры, паттерны).

**Что мы сделали**: заменили `background-image` в hero-слайдере на `<img>`:

```vue
<!-- Было (baseline): -->
<div :style="{ backgroundImage: `url(${slide.image})` }">

<!-- Стало: -->
<img :src="slide.image" class="hero-slider__image"
     width="1280" height="500" :fetchpriority="..." :loading="...">
```

С CSS `object-fit: cover` для правильного масштабирования:

```css
.hero-slider__image {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

### Возможные вопросы аудитории

**В: "`fetchpriority` поддерживается везде?"**

О: Chrome 101+, Edge 101+, Safari 17.2+, Firefox 132+. Но это **progressive enhancement** -- если браузер не поддерживает атрибут, он просто его игнорирует. Картинка загрузится как обычно. Нет никакого риска в добавлении этого атрибута.

**В: "Можно ли lazy loading через JS вместо атрибута?"**

О: Можно -- через Intersection Observer. Но нативный `loading="lazy"` имеет преимущества: он работает **до выполнения JavaScript** (парсер обрабатывает его сразу), интегрирован с системой приоритетов браузера и не требует дополнительного кода. Нативный вариант всегда лучше, если не нужна кастомная логика (например, placeholder/blur-up эффект).

**В: "А как насчёт WebP/AVIF?"**

О: Форматы следующего поколения -- важная оптимизация, но **ортогональная** к тому, что мы рассматриваем. `fetchpriority` и `loading="lazy"` работают с любым форматом. Форматы уменьшают размер файлов, а мы оптимизируем **приоритеты и порядок загрузки**. Лучший результат -- когда применяете обе техники вместе.

**В: "Зачем `decoding="async"`?"**

О: Декодирование изображения (из JPEG/PNG в пиксели для рендеринга) происходит на main thread и может блокировать его на 5-20 мс для каждой большой картинки. `decoding="async"` разрешает браузеру декодировать в фоновом потоке. Не все браузеры это поддерживают, но опять же -- это progressive enhancement.

## Итерация 3: Рендеринг и CSS

**Ветка:** `iter-3-rendering`
**Изменённые файлы:** `index.html`, `ProductCard.vue`, `ProductsSection.vue`, `ReviewsList.vue`, `AboutSection.vue`
**Lighthouse:** Score 80 -> 82, LCP 4851 -> 4694 мс (-3%), TBT 17.5 -> 10.5 мс (-40%)

### Объяснение проблемы

#### Critical Rendering Path

```
HTML → DOM
                 ↘
                   Render Tree → Layout → Paint → Composite → Пиксели на экране
                 ↗
CSS → CSSOM
```

**Ключевая деталь**: браузер **не рисует ни одного пикселя**, пока не построит CSSOM. Внешний CSS-файл -- это render-blocking ресурс. Даже если DOM построен, пользователь видит белый экран, пока CSS не скачается и не распарсится.

**Вторая проблема**: браузер по умолчанию вычисляет layout и paint для **всех элементов на странице**, включая те, что находятся на 5000 пикселей ниже viewport. Это лишняя работа для main thread.

### Inline Critical CSS

#### Что такое "critical CSS"

Critical CSS -- это **минимальный набор стилей**, необходимый для рендеринга контента выше фолда (первый экран без скролла).

Для нашего сайта это:
- CSS-переменные (`:root {}`)
- Базовый reset (`*, *::before, *::after {}`)
- Body styles
- Header styles (фиксированный хедер виден всегда)
- Hero-слайдер (первый экран)
- `@font-face` правила (чтобы шрифт начал загружаться сразу)

#### Как это работает

```html
<head>
  <!-- Critical CSS встроен прямо в HTML — не нужен сетевой запрос -->
  <style>
    :root { --color-primary: #2563eb; ... }
    * { box-sizing: border-box; margin: 0; }
    body { font-family: 'Inter', sans-serif; }
    .header { position: fixed; ... }
    .hero-slider { height: 500px; ... }
    @font-face { font-family: 'Inter'; font-display: swap; src: url(...); }
  </style>

  <!-- Остальной CSS загружается асинхронно -->
  <link rel="stylesheet" href="/vendor/leaflet/leaflet.css">
</head>
```

**Почему это помогает FCP**: браузер может нарисовать первый экран **сразу после парсинга HTML**, не дожидаясь загрузки внешних CSS-файлов. Все стили для первого экрана уже в HTML.

#### Трейдоффы

| Плюс | Минус |
|------|-------|
| Мгновенный FCP (нет ожидания CSS) | Увеличивает размер HTML |
| Работает даже при медленном CDN | Стили не кешируются отдельно |
| Не зависит от приоритетов загрузки | Нужно поддерживать два набора стилей |

Для first visit (новый пользователь) inline CSS **всегда выигрывает**. Для repeat visit (стили закешированы) разница минимальна, но HTML всё равно чуть больше.

**Что мы сделали**: встроили critical CSS в `<head>` файла `index.html`. Это минифицированные стили для хедера, hero-слайдера и базового layout. Убрали внешний `fonts/inter/inter.css` -- `@font-face` теперь внутри inline-стилей.

### content-visibility: auto

#### Что делает

```css
#products {
  content-visibility: auto;
  contain-intrinsic-size: auto 2000px;
}
```

`content-visibility: auto` говорит браузеру: **"Не рендери содержимое этого элемента, пока он не приблизится к viewport"**.

#### Как работает внутри

Без `content-visibility`:
```
Парсер построил DOM → Браузер вычисляет layout ВСЕХ 50 карточек товаров
                    → Рисует ВСЕ 50 карточек
                    → Пользователь видит только 6 из них
```

С `content-visibility: auto`:
```
Парсер построил DOM → Браузер вычисляет layout только видимых элементов
                    → Рисует только ~6 карточек в viewport
                    → Остальные пропущены (layout, paint, style skipped)
                    → При скролле: рендерит по мере приближения к viewport
```

Для страницы с 50 карточками: вместо рендеринга 50 элементов рендерятся ~6. Это **88% экономии** работы рендеринга.

#### contain-intrinsic-size

Когда элемент с `content-visibility: auto` находится за пределами viewport, его rendered height = 0. Это ломает скроллбар -- он будет прыгать при скролле.

`contain-intrinsic-size: auto 2000px` говорит браузеру: "Считай, что этот элемент занимает ~2000 px, пока не узнаешь реальный размер". Ключевое слово `auto` запоминает реальный размер после первого рендеринга.

**Что мы сделали**: добавили `content-visibility: auto` на секции, которые находятся ниже фолда:

```css
/* Секция товаров (ниже hero-слайдера) */
#products {
  content-visibility: auto;
  contain-intrinsic-size: auto 2000px;
}

/* Секция отзывов */
#reviews {
  content-visibility: auto;
  contain-intrinsic-size: auto 600px;
}

/* Секция "О нас" */
#about {
  content-visibility: auto;
  contain-intrinsic-size: auto 1200px;
}
```

### contain: content

#### CSS Containment

CSS Containment -- механизм изоляции. Когда вы меняете что-то внутри элемента (текст, размер картинки, добавляете child), браузер по умолчанию **пересчитывает layout всей страницы** (reflow). Это дорого.

`contain: content` -- сокращение для `contain: layout paint style`:

| Тип | Что делает |
|-----|-----------|
| `layout` | Изменения внутри элемента не вызывают reflow родителей и соседей |
| `paint` | Дочерние элементы не рисуются за пределами границ контейнера |
| `style` | CSS-счётчики и другие style-эффекты изолированы |

#### Аналогия

Без `contain`:
```
Изменилась карточка товара #5
  → Пересчитать layout карточки #5
  → Пересчитать layout grid-контейнера
  → Пересчитать layout секции
  → Пересчитать layout всей страницы
```

С `contain: content`:
```
Изменилась карточка товара #5
  → Пересчитать layout ТОЛЬКО карточки #5
  → Готово. Ничего больше не трогаем.
```

Каждая карточка -- в своём "песочнице". Изменения изолированы.

**Что мы сделали**: добавили `contain: content` на карточки товаров:

```css
.product-card {
  contain: content;
  /* ... остальные стили ... */
}
```

## Итерация 4: Серверные оптимизации

**Ветка:** `iter-4-server`
**Изменённые файлы:** `server.js`, `package.json`
**Lighthouse:** Score 82 -> 99, FCP 1955 -> 1206 мс (-38%), LCP 4694 -> 2066 мс (-56%)

### Объяснение проблемы

Даже с идеально оптимизированным фронтендом, скорость **доставки** ресурсов критична:

- **HTTP/1.1**: максимум 6 TCP-соединений на домен. 30 файлов = 5 раундов ожидания. Head-of-line blocking: медленный файл блокирует все остальные в том же соединении.
- **Без сжатия**: файлы передаются в полном размере. JavaScript 200 КБ так и передаётся как 200 КБ.
- **Без кеширования**: каждый визит скачивает все файлы заново. Возврат через 5 минут -- всё заново.
- **Последовательность запросов**: браузер узнаёт о sub-ресурсах (CSS, JS, шрифты) только из HTML. А HTML приходит только после обработки на сервере.

### HTTP/2 Multiplexing

#### Проблема HTTP/1.1

```
Соединение 1: [file1]───────[file7]───────[file13]
Соединение 2: [file2]───────[file8]───────[file14]
Соединение 3: [file3]───────[file9]───────[file15]
Соединение 4: [file4]───────[file10]──────[file16]
Соединение 5: [file5]───────[file11]──────...
Соединение 6: [file6]───────[file12]──────...
              │             │              │
              Раунд 1       Раунд 2        Раунд 3
```

Максимум 6 параллельных загрузок. Если `file1` медленный -- `file7` ждёт (head-of-line blocking).

#### Решение HTTP/2

```
Одно TCP-соединение:
  Stream 1: [file1]
  Stream 2: [file2]
  Stream 3: [file3]
  ...
  Stream 30: [file30]
  ← Все файлы загружаются одновременно через одно соединение →
```

HTTP/2 использует **бинарное фреймирование**: одно TCP-соединение, множество потоков (streams). Каждый файл -- отдельный поток. Потоки мультиплексируются: данные чередуются на уровне фреймов.

**Head-of-line blocking решён** на уровне HTTP (но остаётся на уровне TCP -- это решает HTTP/3 с QUIC).

**Требование**: HTTP/2 работает только поверх HTTPS (TLS). Поэтому для демо нам нужен self-signed сертификат.

**Что мы сделали**: заменили `http.createServer` на `http2.createSecureServer`:

```javascript
import { createSecureServer } from 'node:http2'

const server = createSecureServer({
  key: readFileSync(keyPath),
  cert: readFileSync(certPath),
  allowHTTP1: true  // фолбэк для старых клиентов
}, app)
```

### Brotli-сжатие

#### Что это

Brotli -- алгоритм сжатия, разработанный Google специально для веба. Главное отличие от gzip: **встроенный словарь** из ~120 КБ, содержащий частые строки веб-контента:

- HTML-теги: `<div>`, `<span>`, `<script>`, `</`, `class="`
- CSS-свойства: `display:`, `margin:`, `font-`, `background-`
- JavaScript-ключевые слова: `function`, `return`, `const`, `export`
- HTTP-заголовки, распространённые URL-паттерны

Благодаря этому словарю Brotli сжимает веб-контент **на 20-30% лучше gzip**.

#### Уровни качества (0-11)

| Уровень | Скорость сжатия | Степень сжатия | Применение |
|---------|----------------|----------------|------------|
| 0-3 | Очень быстро | Как gzip | Нет смысла использовать Brotli |
| **4** | Быстро | Лучше gzip на 15-20% | **Динамический контент** (real-time) |
| 5-9 | Средне | Хорошо | Компромисс |
| **11** | Медленно (секунды) | Максимум (-30% vs gzip) | **Статические файлы** (однократная компрессия) |

**Наш подход**: quality 4 для динамического сжатия в real-time (HTML, ответы API):

```javascript
const brotli = createBrotliCompress({
  params: { [constants.BROTLI_PARAM_QUALITY]: 4 }
})
pipeline(createReadStream(filePath), brotli, res, () => {})
```

В production-окружении статические файлы (JS, CSS) лучше сжимать заранее на уровне 11 при build (через vite-plugin-compression или аналог).

### Стратегия Cache-Control

#### Три уровня кеширования

**1. HTML (`no-cache`)**

```
Cache-Control: no-cache
```

`no-cache` -- **не означает "не кешировать"**! Означает: "кешируй, но перед использованием проверь с сервером". Браузер отправляет запрос с `If-Modified-Since` или `If-None-Match`. Сервер отвечает:
- `304 Not Modified` (без тела) -- используй кеш. Быстро.
- `200` с новым контентом -- HTML изменился, вот новая версия.

Почему `no-cache` для HTML: HTML содержит ссылки на JS/CSS-файлы с хешами. Если мы задеплоили обновление -- хеши поменялись, и HTML должен указывать на новые файлы. Поэтому всегда проверяем актуальность.

**2. JS/CSS с хешем (`immutable, max-age=31536000`)**

```
Cache-Control: public, max-age=31536000, immutable
```

Vite генерирует имена файлов с content hash: `index-CwvPyqso.css`. Если содержимое файла изменится -- хеш изменится -- URL изменится. Старый файл в кеше **никогда не будет запрошен** по новому URL.

- `max-age=31536000` -- кешировать на 1 год (365 дней).
- `immutable` -- не отправлять даже revalidation-запросы. Браузер не будет проверять "может файл изменился?" -- он знает, что immutable файл **не может измениться** (потому что хеш в имени).

**3. Картинки (`max-age=86400`)**

```
Cache-Control: public, max-age=86400
```

Кешировать на 24 часа. Картинки не имеют хешей в именах, поэтому нельзя кешировать навсегда -- при обновлении картинки пользователь увидит изменения максимум через сутки.

**Что мы сделали**: добавили middleware для Cache-Control:

```javascript
app.use((req, res, next) => {
  const url = req.url

  if (url.match(/\.(js|css)$/) && url.includes('-')) {
    // Хешированные ассеты — иммутабельный кеш
    res.set('Cache-Control', 'public, max-age=31536000, immutable')
  } else if (url.match(/\.(jpg|jpeg|png|gif|webp|avif|svg|ico|woff2?)$/)) {
    // Картинки и шрифты — кеш на 24 часа
    res.set('Cache-Control', 'public, max-age=86400')
  } else {
    // HTML — всегда проверять свежесть
    res.set('Cache-Control', 'no-cache')
  }
  next()
})
```

### 103 Early Hints

#### Проблема

```
Обычный flow:
Браузер ────запрос──→ Сервер обрабатывает (500 мс) ──→ 200 + HTML
                      ← Браузер простаивает →
                                                        Парсит HTML...
                                                        Находит CSS/JS/шрифты...
                                                        Начинает загрузку ресурсов
```

Между отправкой запроса и получением HTML проходит 200-500 мс (обработка на сервере, шаблонизация, запросы к БД). Всё это время браузер **ничего не делает**.

#### Решение: 103 Early Hints

```
Браузер ────запрос──→ Сервер СРАЗУ отвечает 103 (ещё до обработки)
                      ← 103 Early Hints:
                         Link: </assets/index.css>; rel=preload; as=style
                         Link: </assets/index.js>; rel=modulepreload
                         Link: </images/hero/hero-1.jpg>; rel=preload; as=image

                      Браузер СРАЗУ начинает загрузку CSS, JS, hero-картинки!

                      ← Тем временем сервер обрабатывает запрос (500 мс) →

                      ← 200 + HTML (ресурсы уже частично загружены!)
```

**Выигрыш**: 200-500 мс. Браузер начинает загрузку критических ресурсов **параллельно** с обработкой на сервере, а не **последовательно** после неё.

**Что мы сделали**: добавили 103 Early Hints в SPA fallback:

```javascript
app.get('*', (req, res) => {
  if (res.writeEarlyHints) {
    res.writeEarlyHints({
      link: [
        `</assets/${cssFile}>; rel=preload; as=style`,
        `</assets/${jsFile}>; rel=modulepreload`,
        '</images/hero/hero-1.jpg>; rel=preload; as=image'
      ]
    })
  }
  // ... отправка HTML
})
```

`res.writeEarlyHints` -- нативный метод Node.js, доступный в HTTP/2 сервере. Мы динамически определяем имена файлов с хешами (читая `dist/assets/`), чтобы Early Hints указывали на правильные файлы.

### Сценарий демо для итерации 4

1. Открыть Network tab. Добавить столбец "Protocol" (правый клик на заголовке таблицы -> Protocol).
2. Загрузить страницу. **Показать**: все запросы используют `h2` (HTTP/2).
3. Кликнуть на JS/CSS-файл. **Показать Response Headers**: `content-encoding: br` (Brotli).
4. Кликнуть на HTML-запрос. **Показать**: `cache-control: no-cache`.
5. Кликнуть на хешированный JS-файл. **Показать**: `cache-control: public, max-age=31536000, immutable`.
6. **Второй заход**: перезагрузить страницу. Показать, что хешированные файлы берутся из кеша (Status: 304 или `(disk cache)`).
7. Lighthouse: **Score 82 -> 99**, LCP 4694 -> 2066 мс (-56%).

### Объяснение метрик

- **Score 82 -> 99 (+17 баллов!)**: Самый большой скачок. Все серверные оптимизации работают вместе: сжатие уменьшает transfer size, HTTP/2 параллелизирует загрузку, кеширование исключает повторные загрузки, Early Hints дают head start.
- **FCP -38% (1955 -> 1206 мс)**: Early Hints + Brotli + HTTP/2 = критические ресурсы (CSS, JS, шрифты) приходят значительно быстрее. Brotli уменьшает CSS на ~30%, HTTP/2 убирает очерёдность, Early Hints начинают загрузку на ~300 мс раньше.
- **LCP -56% (4694 -> 2066 мс)**: Hero-картинка выигрывает от HTTP/2 multiplexing (не конкурирует за TCP-соединения), Brotli-сжатие HTML приходит быстрее (а вместе с ним и ссылка на hero), закешированные ресурсы не забивают bandwidth.
- **SI -52% (2491 -> 1206 мс)**: Вся страница визуально заполняется в два раза быстрее.

### Возможные вопросы аудитории

**В: "HTTP/2 Server Push не лучше Early Hints?"**

О: HTTP/2 Server Push был **deprecated в Chrome 106** (октябрь 2022). Проблема Push: сервер отправлял ресурсы, не зная, что у браузера уже есть в кеше. Если файл закеширован -- Push тратил bandwidth впустую. Early Hints решают эту проблему: сервер только **подсказывает**, что нужно загрузить, а браузер сам решает -- качать или использовать кеш.

**В: "Brotli медленнее gzip при компрессии?"**

О: На уровне 11 -- да, **значительно** медленнее (может занять секунды для большого файла). Поэтому стратегия: статические файлы (JS, CSS) сжимаем заранее при сборке на уровне 11 (один раз сжал -- миллион раз отдал). Динамический контент (HTML) сжимаем на лету на уровне 4, который быстрее gzip и даёт лучшее сжатие.

**В: "`Cache-Control: immutable` -- а если баг?"**

О: Файл имеет хеш в имени: `index-CwvPyqso.js`. Если вы деплоите фикс, содержимое файла меняется -> хеш меняется -> URL становится `index-Xk7mNp2Q.js`. Браузер запрашивает **новый** URL. Старый файл может лежать в кеше навечно -- это нормально, его URL больше никто не запросит. Именно поэтому content hash в именах файлов делает aggressive caching безопасным.

**В: "103 Early Hints поддерживается в Nginx?"**

О: Nginx: нативная поддержка с версии 1.25.3+ (декабрь 2023). До этого -- через сторонний модуль `ngx_http_early_hints`. Apache: через `mod_http2`. Node.js: нативно через `res.writeEarlyHints()` (мы так и сделали). Cloudflare и Fastly поддерживают Early Hints на уровне CDN.

**В: "Self-signed сертификат -- это ок для демо?"**

О: Для демо -- да. В продакшене используйте Let's Encrypt (бесплатно). Браузер покажет предупреждение о self-signed сертификате -- просто нажмите "Advanced -> Proceed". Для Lighthouse это не влияет на результат.
